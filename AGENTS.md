# Workflow для AI-агентов в этом репозитории

Этот файл — обязательное чтение для любого AI-агента, который начинает
работу с кодом проекта. Читать ПЕРВЫМ, ещё до чтения задачи.

## Цикл изменений: локалка → git → прод → CRM-проверка

Любая правка (баг или фича) проходит **строго в этом порядке**. Не
прыгать сразу на прод, не «тестировать на проде вместо локалки».

### 1. Локалка — root cause + fix

- Найти причину в коде или БД (read, grep, репро). Не «починим что-то
  похожее» — именно root cause.
- Починить локально (или написать миграцию в `supabase/migrations/`).
- `npm run typecheck` в `work/` — должно пройти без ошибок.
- Если изменение во фронте — мысленно прокрутить базовый сценарий или
  поднять локальный dev-сервер.
- Не коммитить, пока локально не убедился, что не сломалось.

### 2. Git

- Один логически связанный коммит с осмысленным сообщением (что и
  **почему**).
- `git push origin main` (или текущая рабочая ветка).

### 3. Live-проверка на проде (`alessannailu.com` + CRM)

Делать **как реальный клиент** в браузере:

1. Открыть прод с cache-bust (`?cb=N`).
2. Пройти полный сценарий заявки:
   - 1 услуга → отправить.
   - Несколько услуг (2–5, пик).
   - Разные мастера: один конкретный / «Не важно» / выбор из общего select.
3. **Создать реальный тестовый заказ** с явной пометкой
   `ТЕСТ-N — automated check, не обрабатывать` в имени или комментарии,
   чтобы салон сразу видел и не звонил клиенту.
4. Затем зайти в CRM как сотрудник и проверить:
   - заказ появился в списках;
   - отображается в календаре (день / неделя / месяц);
   - корректно связан с мастером (или «Не важно»);
   - источник (`source`) и `created_by_staff_id` записаны корректно;
   - повторить для разных мастеров (1 / несколько / все доступные).

### 4. Если что-то не сработало

Возвращаемся в шаг 1. Не «допиливаем» прямо на проде через консоль или
SQL-патчи руками — только через миграцию + git.

## Чего НЕЛЬЗЯ

- ❌ Тестировать только на проде, пропустив локалку.
- ❌ Накатывать SQL руками в Supabase Studio мимо `supabase/migrations/`.
  Если применил миграцию через MCP `apply_migration` — обязательно
  положи тот же SQL в `supabase/migrations/NNN_*.sql` в репо.
- ❌ Push без `npm run typecheck`.
- ❌ Создавать тестовые заказы без пометки `ТЕСТ-N` (салон будет звонить
  «клиенту»).
- ❌ Считать прогон законченным без проверки в CRM (списки + календарь).

## Структура репо (вкратце)

- `index.html`, `script.js`, `style.css` — публичный сайт салона
  (`alessannailu.com`, статика, GitHub Pages).
- `work/` — React-приложение CRM (Vite + Tailwind, i18n: ru/et).
- `supabase/migrations/NNN_*.sql` — миграции БД, нумерация по порядку.
  Все DDL только через миграцию.
- `locales/{ru,et,en,fi}.json` — переводы. CRM использует только
  `ru` и `et`; `en`/`fi` — для публичного сайта.
- `server/` — лёгкий Node-fallback API (используется редко, основной
  поток через Supabase RPC).

## Источник записей в `appointments` (миграция 053)

- `source`: `public_site` | `reception` | `crm`. Default = `crm`.
- `created_by_staff_id`: uuid сотрудника, который физически нажал
  «создать запись» в CRM или на ресепшене. NULL для public_site.
- Все новые места создания записей **обязаны** проставлять source
  явно (через RPC `public_book_chain` есть параметр `p_source` +
  `p_created_by_staff_id`).
