# Multi-location (несколько салонов) — техническое предложение

Статус: **черновик / не реализовано**, ждёт отмашки на разработку.
Дата: 2026-04-20.

## 1. Цель

Поддержать в одной CRM/витрине несколько физических точек:
- разные адреса, телефоны, расписания работы,
- свои мастера и услуги (с возможностью «общих» мастеров между точками),
- свой календарь и ресурсы,
- единая база клиентов и единый список ролей сотрудников,
- сводная аналитика по сети + аналитика по каждой точке.

## 2. Минимальный объём (MVP)

1. Сущность `locations` (id, name, slug, address, timezone, phone, email, is_active).
2. У `appointments`, `staff_schedules`, `time_off`, `service_listings`, `inventory_items`,
   `email_jobs`, `outbox_messages` появляется `location_id` (NULLable на старте,
   потом backfill + NOT NULL).
3. У `staff` — отношение many-to-many: `staff_locations(staff_id, location_id, is_primary)`.
4. UI:
   - переключатель локации в шапке (для админов/менеджеров),
   - страница `/admin/locations` для CRUD,
   - календарь, расписания, склад фильтруются по `current_location_id`.
5. RPC `current_location_id()` (на сессию через GUC `app.current_location_id` или localStorage),
   используется во всех RLS-политиках.

## 3. Расширения (фаза 2)

- Публичный сайт: домены/поддомены `tallinn.salonai.ee`, `parnu.salonai.ee` или query `?location=parnu`.
- Расписание работы салона по локации (часы открытия), а не глобальное.
- Прайс может отличаться по локации (`service_listings.location_id NULL` = глобальный, иначе override).
- Inventory — склад на каждую точку (`inventory_items.location_id` обязательное).
- Email-шаблоны — поддержка переменных `{{location_name}}`, `{{location_address}}`.

## 4. Аналитика

- Все view-аналитики (`analytics_daily`, `analytics_top_services_*`, `analytics_kpi_now`)
  получают необязательный фильтр по `location_id` (вторая версия view с GROUP BY,
  либо добавить колонку `location_id` и в дашборде делать `where location_id = ?`).
- На дашборде в шапке — селектор «Все локации / Pärnu / Tallinn».

## 5. RLS-стратегия

- Новый GUC `app.current_location_id` ставится фронтом при логине / переключении локации.
- Политика чтения: `location_id = current_location_id() OR is_admin()`.
- Запись: только если у сотрудника есть `staff_locations(staff_id, location_id)`.

## 6. Миграция данных

1. `insert into locations(name, slug) values ('Главный салон', 'main') returning id;`
2. Backfill: всем существующим записям подставить `location_id = main_id`.
3. Один спринт «двойного режима»: NULL = считается main.
4. После проверки — `NOT NULL` + RLS с обязательным фильтром.

## 7. Оценка трудозатрат

- Миграции БД + RLS: **~1 день**.
- UI переключатель + admin/locations: **~0.5 дня**.
- Перевод календаря/склада/писем на location_id: **~1.5 дня**.
- Multi-domain публичного сайта (опционально): **~1 день**.
- **Итого MVP: 3 дня инженерных работ.**

## 8. Чего НЕ делаем в MVP

- Отдельные подписки/тарификация по локациям.
- Биллинг между локациями (трансферы клиентов).
- Сложное распределение мастеров «гастролёры по графику».
  Эти задачи откладываются до реального второго салона.

## 9. Что нужно от заказчика для старта

- Подтверждение списка локаций и их базовых данных (имя, адрес, телефон, email, TZ).
- Решение: общий список услуг или у каждой локации свой?
- Решение: общий склад или раздельный?
- Решение по доменам: один домен с переключателем или поддомены?
