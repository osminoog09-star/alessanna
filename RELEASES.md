# Releases — версии и стабильные точки

> **Зачем этот файл.** В проекте две живые системы (публичный сайт `alessannailu.com` и CRM `work.alessannailu.com`) и обе деплоятся из `main`. Чтобы каждый раз можно было быстро откатиться, мы помечаем **стабильные** коммиты git-тегами и фиксируем их здесь.
>
> Подробные патчноуты — в [`CHANGELOG.md`](./CHANGELOG.md). Этот файл короткий и чисто оперативный.

---

## Правила работы

1. **Стабильная точка** = коммит, который полностью прошёл live-проверку на проде (CRM и/или сайт) и можно безопасно деплоить ещё раз.
2. На стабильный коммит ставится тег `stable-YYYY-MM-DD-<scope>`, где `<scope>` — короткий ярлык (`crm-ux`, `public-site-cleanup`, `crm-bookings`, …).
3. Пишем описание тега через `git tag <name> <sha> -m "..."` — оно остаётся в репо.
4. После пуша в `main` рядом с тегом обновляем секцию ниже (одна строка с датой, тегом, sha, ссылкой на коммит и кратким «что внутри»).
5. **Откат:** `git checkout <tag>` или для полного rollback в `main` — `git revert <range>` и пуш отдельным коммитом. Force-push в `main` не делаем.
6. Экспериментальные/большие правки — в отдельных ветках (`feature/*`, `cursor/*`). В `main` мёрджим только когда зелёный билд и пройден ручной чек.
7. **«В работе» состояние** = всё, что есть в `main` поверх последнего стабильного тега. Не считается готовым к продакшен-стрессу, пока не получило новый `stable-*` тег.

### Правило техники безопасности (с 2026-04-19)

Любая правка, которая может затронуть пользовательский поток или данные, проходит проверку **в этом порядке**:

1. **Локальная сборка/среда** — `http-server` для статического сайта + `vite/next dev` для CRM (`work/`).
2. **Локальный smoke** клиентского сценария (browser-use или вручную): прайс → корзина → форма → подтверждение. Если правка про БД/RPC — обязательно прогнать сценарий, который реально дергает Supabase (`public_book_chain`, RLS, миграции).
3. **CRM/база** — отдельно проверить admin/master/manager-роли, что новая фича/фикс ничего не сломал в `/admin/*` и в `staff_services`/`bookings`/RLS-политиках. Если правились миграции — `supabase migration up --dry-run` локально.
4. **Только после** локальной зелёной картины — мёрдж в `main`, пуш, тег `stable-*`, и `vercel`/`gh-pages` подтянет на прод.
5. На проде — повторить тот же smoke глазами (как клиент и как админ). Если появилась regression — откат через тег предыдущего `stable-*`.

«Рискованное» = всё, что трогает RPC/RLS/миграции, форму записи, таблицу `bookings`, расписание и видимость карточек мастеров/услуг публичного сайта.

## Как добавить новый стабильный релиз

```bash
# 1. убедились что main зелёный (build CRM, базовый smoke публичного сайта)
cd work && npx tsc --noEmit && npm run build && cd ..

# 2. поставили тег
git tag stable-2026-04-25-crm-staff -m "Stable: разделили AdminStaffPage на под-экраны"

# 3. запушили тег (один)
git push origin stable-2026-04-25-crm-staff

# 4. дописали строку в таблицу ниже + обновили CHANGELOG
```

Откат до конкретного релиза:

```bash
git checkout stable-2026-04-19-crm-ux        # просто посмотреть
git checkout -b hotfix/from-stable stable-... # ветка от стабильной точки
```

## В работе (`main` ↑ от последнего стабильного)

- Поддержка `/admin/support`: «закрепить за собой / снять / передать», display_id (SAL/SIT/EMP), client_ip + device fingerprint + флаг «⚠ Подозрительно», stats-дашборд (5 плиток), новая миграция `038_support_assignment_ids_ip_stats.sql` (применена на проде 4 транзакциями `038`/`038b`/`038c`/`038d` через MCP). БД-RPC прогнаны smoke'ом против прода. Live UI smoke ждёт владельца (нужен PIN).
- `/admin/services` UX: компактный «аккордеон» услуг (по дефолту строка-сводка `название · цена · мин · мастера · ⚠/на сайте`, клик раскрывает полный редактор), сворачиваемые секции категорий, новая полоса фильтров (статус/«без мастеров»/«не на главной»/категории/сортировка), кнопки «Развернуть/Свернуть всё», запоминание состояния в `localStorage` (`admin/services/v1`). Без миграций и без новых RPC. tsc + build green. Live UI smoke ждёт владельца на проде.
